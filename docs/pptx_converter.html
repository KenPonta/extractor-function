.pptx — exact

Each picture is a shape on a slide, so images are placed inline during parsing.

  • parser: python-pptx
  • placement: exact, in reading order
  • marker: [Image N]
  • skips stage 4

.ppt — inferred

The image→slide link is lost. Images are registered in deck order, described, then matched to slides by the model.

  • parser: sharepoint2text
  • placement: LLM match (else proportional spread)
  • marker: [Image N (approximate slide)]
  • order clamped non-decreasing

The three clever bits

Dedup by pixels, not bytes

content_digest decodes each image and hashes its pixels. .ppt re-encodes the same picture per slide; pixel-hashing collapses those into one image described once.

Deferred, inferred .ppt placement

.ppt loses image→slide links, so images are registered in deck order, described, then match_slides assigns each to a slide — with a proportional spread as fallback.

Trust-but-verify ordering

The model suggests a slide per figure, but the code clamps assignments non-decreasing and snaps them to real slides — a bad reply can lower quality but can't corrupt the deck.
